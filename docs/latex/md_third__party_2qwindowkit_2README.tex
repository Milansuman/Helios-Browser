 Cross-\/platform window customization framework for Qt Widgets and Qt Quick.

This project inherited most of \href{https://github.com/wangwenx190/framelesshelper}{\texttt{ wangwenx190 Frameless\+Helper}} implementation, with a complete refactoring and upgrading of the architecture.

\hypertarget{md_third__party_2qwindowkit_2README_autotoc_md102}{}\doxysubsubsection{\texorpdfstring{Setup Window Agent}{Setup Window Agent}}\label{md_third__party_2qwindowkit_2README_autotoc_md102}
First, setup {\ttfamily Widget\+Window\+Agent} for your top {\ttfamily QWidget} instance. (Each window needs its own agent.)


\begin{DoxyCode}{0}
\DoxyCodeLine{\ ++}
\DoxyCodeLine{\textcolor{preprocessor}{\#include\ <QWKWidgets/widgetwindowagent.h>}}
\DoxyCodeLine{}
\DoxyCodeLine{MyWidget::MyWidget(QWidget\ *parent)\ \{}
\DoxyCodeLine{\ \ \ \ \textcolor{comment}{//\ ...}}
\DoxyCodeLine{\ \ \ \ \textcolor{keyword}{auto}\ agent\ =\ \textcolor{keyword}{new}\ \mbox{\hyperlink{classQWK_1_1WidgetWindowAgent}{QWK::WidgetWindowAgent}}(\textcolor{keyword}{this});}
\DoxyCodeLine{\ \ \ \ agent-\/>setup(\textcolor{keyword}{this});}
\DoxyCodeLine{\ \ \ \ \textcolor{comment}{//\ ...}}
\DoxyCodeLine{\}}

\end{DoxyCode}


If you don\textquotesingle{}t want to derive a new widget class or change the constructor, you can initialize the agent after the window constructs.


\begin{DoxyCode}{0}
\DoxyCodeLine{\ ++}
\DoxyCodeLine{\textcolor{keyword}{auto}\ w\ =\ \textcolor{keyword}{new}\ MyWidget();}
\DoxyCodeLine{\textcolor{keyword}{auto}\ agent\ =\ \textcolor{keyword}{new}\ \mbox{\hyperlink{classQWK_1_1WidgetWindowAgent}{QWK::WidgetWindowAgent}}(w);}
\DoxyCodeLine{agent-\/>setup(w);}

\end{DoxyCode}


You should call {\ttfamily \doxylink{classQWK_1_1WidgetWindowAgent_aaf4c32239222367230bd6c1baa0ff3ae}{QWK\+::\+Widget\+Window\+Agent\+::setup()}} as early as possible, especially when you need to set the size constrains. QWindow\+Kit will change some Qt internal data which will affect how Qt calculates the window size, and thus you need to let QWindow\+Kit initialize at the very beginning.\hypertarget{md_third__party_2qwindowkit_2README_autotoc_md103}{}\doxysubsubsection{\texorpdfstring{Construct Title bar}{Construct Title bar}}\label{md_third__party_2qwindowkit_2README_autotoc_md103}
Then, construct your title bar widget, without which the window lacks the basic interaction feature, and it\textquotesingle{}s better to put it into the window\textquotesingle{}s layout.

You can use the \href{examples/shared/widgetframe/windowbar.h}{\texttt{ {\ttfamily Window\+Bar}}} provided by {\ttfamily Widget\+Frame} in the examples as the container of your title bar components.

Let {\ttfamily Widget\+Window\+Agent} know which widget the title bar is.


\begin{DoxyCode}{0}
\DoxyCodeLine{\ ++}
\DoxyCodeLine{agent-\/>setTitleBar(myTitleBar);}

\end{DoxyCode}


Next, set system button hints to let {\ttfamily Widget\+Window\+Agent} know the role of the child widgets, which is important for the Snap Layout to work.


\begin{DoxyCode}{0}
\DoxyCodeLine{\ ++}
\DoxyCodeLine{agent-\/>setSystemButton(QWK::WindowAgentBase::WindowIcon,\ myTitleBar-\/>iconButton());}
\DoxyCodeLine{agent-\/>setSystemButton(QWK::WindowAgentBase::Minimize,\ myTitleBar-\/>minButton());}
\DoxyCodeLine{agent-\/>setSystemButton(QWK::WindowAgentBase::Maximize,\ myTitleBar-\/>maxButton());}
\DoxyCodeLine{agent-\/>setSystemButton(QWK::WindowAgentBase::Close,\ myTitleBar-\/>closeButton());}

\end{DoxyCode}


Doing this does not mean that these buttons\textquotesingle{} click events are automatically associated with window actions, you still need to manually connect the signals and slots to emulate the native window behaviors.

On mac\+OS, this step can be skipped because it is better to use the buttons provided by the system.

Last but not least, set hit-\/test visible hint to let {\ttfamily Widget\+Window\+Agent} know which widgets are willing to receive mouse events.


\begin{DoxyCode}{0}
\DoxyCodeLine{\ ++}
\DoxyCodeLine{agent-\/>setHitTestVisible(myTitleBar-\/>menuBar(),\ \textcolor{keyword}{true});}

\end{DoxyCode}


The rest region within the title bar will be regarded as the draggable area for the user to move the window, and thus any QWidgets inside it will not receive any user interaction events such as mouse events/focus events/etc anymore, but you can still send/post such events to these widgets manually, either through Qt API or system API.


\begin{DoxyItemize}
\item If you want to disable window maximization, you can remove the {\ttfamily Qt\+::\+Window\+Maximize\+Button\+Hint} flag from the window.

\hypertarget{md_third__party_2qwindowkit_2README_autotoc_md109}{}\doxysubsubsection{\texorpdfstring{Qt Version}{Qt Version}}\label{md_third__party_2qwindowkit_2README_autotoc_md109}

\begin{DoxyItemize}
\item To achieve better frameless functionality, QWindow\+Kit depends heavily on Qt\textquotesingle{}s internal implementation. However, there are many differences in different versions of Qt, and earlier versions of Qt5 and Qt6 have many bugs which make it extremely difficult for QWindow\+Kit to workaround without changing the Qt source code.
\item And also due to limited manpower, although QWindow\+Kit can be successfully compiled on Qt 5.\+12 or later, it can hardly work perfectly on all Qt versions.
\item Therefore, the following Qt version ranges are recommended, if there are any exceptions with QWindow\+Kit in your application, make sure the Qt version you use is in the ranges before raising the issue.
\begin{DoxyItemize}
\item Qt 5\+: 5.\+15.\+2 or higher
\item Qt 6\+: 6.\+6.\+2 or higher (the newer, the better)
\end{DoxyItemize}
\end{DoxyItemize}\hypertarget{md_third__party_2qwindowkit_2README_autotoc_md110}{}\doxysubsubsection{\texorpdfstring{Hot Switch}{Hot Switch}}\label{md_third__party_2qwindowkit_2README_autotoc_md110}

\begin{DoxyItemize}
\item Once you have made the window frameless, it will not be able to switch back to the system frame again unless you destroy your window and recreate it with different settings.
\end{DoxyItemize}\hypertarget{md_third__party_2qwindowkit_2README_autotoc_md111}{}\doxysubsubsection{\texorpdfstring{Native Child Widget}{Native Child Widget}}\label{md_third__party_2qwindowkit_2README_autotoc_md111}

\begin{DoxyItemize}
\item There {\bfseries{must not}} be any internal child widget with {\ttfamily Qt\+::\+WA\+\_\+\+Native\+Window} property enabled, otherwise the native features and display may be abnormal. Therefore, do not set any widget that has called {\ttfamily QWidget\+::win\+Id()} or {\ttfamily QWidget\+::set\+Attribute(\+Qt\+::\+WA\+\_\+\+Native\+Window)} as a descendant of a frameless window.
\begin{DoxyItemize}
\item If you really need to move widgets between different windows, make sure that the widget is not a top-\/level window and wrap it with a frameless container window.
\end{DoxyItemize}
\end{DoxyItemize}\hypertarget{md_third__party_2qwindowkit_2README_autotoc_md112}{}\doxysubsubsection{\texorpdfstring{Size Constrains}{Size Constrains}}\label{md_third__party_2qwindowkit_2README_autotoc_md112}

\begin{DoxyItemize}
\item If you want to disable window resizing, you can set a fixed size, which is officially supported by QWindow\+Kit. If you use other special means to achieve this, \doxylink{namespaceQWK}{QWK} doesn\textquotesingle{}t guarantee everything can still be fully functional.
\item If you set a maximized width or height, the window should not be maximized because you cannot get the correct window size through Qt APIs. You may workaround this by using system APIs such as {\ttfamily Get\+Window\+Rect} or {\ttfamily Get\+Client\+Rect}. The root cause lies deep in Qt QPA implementations and currently we don\textquotesingle{}t know how to fix it without modifying Qt itself.
\end{DoxyItemize}\hypertarget{md_third__party_2qwindowkit_2README_autotoc_md113}{}\doxysubsubsection{\texorpdfstring{Windows 10}{Windows 10}}\label{md_third__party_2qwindowkit_2README_autotoc_md113}

\begin{DoxyItemize}
\item Due to the inherent defects in the Windows 10 window system, the top border will disappear when the system title bar is removed. We have filtered Qt\textquotesingle{}s event and perfectly reshown the system top border, thanks to the implementation of Windows Terminal for our reference. However, this workaround only works with Qt\+Widgets and Qt\+Quick ({\bfseries{only when rendering through D3D}}) applications.
\item In Qt\+Quick applications that use Open\+GL or other rendering backends, we use Qt\textquotesingle{}s painting system to emulate this border. But since Windows 10 system border is translucent, the difference from the system border is more noticeable in a dark background.
